# SnapFlow

> A QA tester was losing 2–4 hours every day to a problem nobody asked her to fix.
> So she fixed it anyway.

---

## The 30-second version

This isn't a college assignment. It's not a hackathon weekend project either.

**I identified a recurring operational inefficiency within an enterprise testing team, then independently designed, built, and evolved a desktop application to solve it—entirely outside my assigned project responsibilities.**

Developed with real tester feedback and refined through day-to-day usage within a live testing environment, SnapFlow was created to eliminate one of the most repetitive parts of functional testing: capturing, organizing, and documenting test evidence.

After demonstrations to project leadership, the solution received positive feedback and sparked discussions around Jira/Xray integration and broader team standardization. Those conversations shifted the project from a personal productivity tool toward a solution designed for wider organizational adoption.

This repository showcases the journey behind SnapFlow—from identifying the business problem to documenting the product vision, architecture decisions, engineering trade-offs, roadmap, and stakeholder impact.

> 🔒 **A note on what you're looking at:** This public repository is intentionally an overview. The complete project documentation—including detailed architecture notes, implementation decisions, roadmap evolution, supporting design artifacts, and technical documentation—lives in a private repository and is available upon request for recruiter and interviewer evaluation.

## The one question worth asking me about this

> *"Why did you choose a desktop application instead of a web application for a tool intended for enterprise testing?"*

That question represents one of the most important design decisions in the entire project. The answer isn't simply about technology—it is about understanding tester workflows, usability, performance, deployment constraints, and selecting the architecture that best supports the people using the product.

The detailed reasoning is documented within the project, but I'd much rather discuss those architectural trade-offs during an interview than let the repository speak entirely for itself.
